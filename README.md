# District Usage Reports

Internal tool for Customer Success Managers to request per-school weekly usage summaries by `district_id` and week-ending date.

## Quick start

### Prerequisites

- Node.js **20+**
- pnpm **10+**

### Install

```bash
pnpm install
```

### Seed the database

```bash
pnpm db:seed
```

This creates a local SQLite database at `data/usage-events.sqlite`.

### Run the dev server

```bash
pnpm dev
```

Open: `http://localhost:3000`

### Run tests

```bash
pnpm test
```

## Demo walkthrough for the reviewer

Use the seeded dataset generated by `pnpm db:seed`.

### Seeded district IDs

- `district-1`
- `district-2`
- `district-3`

### Week-ending date that works

- `2026-05-30`

The seeded data spans:
- earliest timestamp: `2026-05-03T00:01:37.444Z`
- latest timestamp: `2026-05-30T23:34:03.408Z`

### Step by step

1. Run `pnpm db:seed`
2. Run `pnpm dev`
3. Open `http://localhost:3000`
4. Enter `district-1` in the `District ID` field
5. Pick `2026-05-30` as the week-ending date
6. Click **Load summary**
7. Review the district summary card:
   - active teachers
   - total events
   - top 3 event types
8. Review the per-school breakdown table
9. Click the **Active teachers** or **Total events** column controls to sort
10. Click **Copy Summary** to copy a plain-text handoff report
11. Press `Ctrl+P` (or `Cmd+P` on macOS) to verify the print-friendly layout

You can also try:
- `district-2` with `2026-05-30`
- `district-3` with `2026-05-30`

## Design choices and trade-offs

### ADR references

- `docs/adr/0001-sqlite-with-better-sqlite3.md`
- `docs/adr/0002-aggregation-strategy-and-date-semantics.md`

### Why SQLite over Postgres

For this assessment, SQLite keeps reviewer setup friction extremely low:

- no Docker
- no cloud database
- no connection string setup
- deterministic local file after `pnpm db:seed`

Trade-off:
- this is not the long-term production database choice
- production would move to Postgres with migrations and operational controls

### Why raw SQL over an ORM

The data model is intentionally small: one `usage_events` table and a narrow query surface.

Raw SQL made sense here because it:
- keeps the aggregation path explicit and easy to defend in a walkthrough
- avoids adding schema-generation and ORM abstraction overhead
- matches the project rule of simplicity over abstraction

Trade-off:
- less compile-time safety than a typed query builder
- more discipline required around query result typing and tests

### Why tRPC

The app is a single Next.js full-stack codebase. tRPC fits well because it:
- keeps the client/server contract type-safe
- avoids a separate REST DTO/codegen layer
- works naturally with the App Router and React Query

Trade-off:
- it is more coupled to the TypeScript app boundary than a public REST API would be

### Why client-side sorting

The table is sorting already-fetched weekly summary rows, not a large paginated dataset.

Client-side sorting is a good fit because it:
- keeps the interaction instant
- avoids extra network requests
- keeps the server focused on the weekly summary itself

Trade-off:
- if this table grew much larger, server-side sorting/pagination would be worth revisiting

### Aggregation semantics

Per ADR-0002, the app uses a **7-day sliding window** ending on the selected date, not an ISO calendar week.

Example:
- selected week-ending date: `2026-05-30`
- computed range: `2026-05-24` through `2026-05-30`
- SQL filter pattern: `timestamp >= weekStarting AND timestamp < weekEndingPlusOne`

This avoids common off-by-one and end-of-day timestamp bugs.

## v2 roadmap

If I kept building this beyond the assessment, I would prioritize:

- migrate from SQLite to Postgres
- add authentication and role-based access
- support CSV ingestion with validation and import reporting
- scheduled report generation
- PDF / CSV export
- caching for repeated historical requests
- multi-district comparison views
- API rate limiting
- observability for request health and usage patterns

## What would change for production

For a production rollout, I would change the system in the following ways:

### Data and schema
- Postgres instead of SQLite
- migration tooling and versioned schema changes
- stronger relational modeling if the domain grows

### Security and platform
- authentication and authorization
- security audit of request boundaries, secrets, and dependencies
- rate limiting and abuse protections
- explicit data retention policies

### Delivery and quality
- CI/CD with GitHub Actions
- automated code quality checks
- AI review assistance such as **CodeRabbit** or **Cursor Bugbot** in PR workflows
- stricter pre-merge verification

### Testing
- broader integration tests around tRPC + database behavior
- E2E coverage with Playwright for the full UI flow
- seeded test fixtures for regression coverage

### Operations
- monitoring and alerting
- structured logs and dashboards
- performance visibility on query latency and error rates

## How I used AI tools

### Tools used

- **Pi** for agentic coding workflows

### Workflow used

The main workflow was:
- propose a change
- generate proposal/design/spec/tasks
- apply the change with TDD-first implementation
- verify with tests/typecheck
- archive the change with spec sync

Important OpenSpec note:
- the safest close-out command is `openspec archive <name> -y`
- that archives the change **and** syncs delta specs into `openspec/specs/`
- if an older `/opsx-archive` prompt is used and it only moves the folder to `changes/archive/`, run `/opsx-sync` immediately after so `openspec/specs/` does not stay empty or stale

### What I delegated to AI

I delegated:
- OpenSpec artifact generation from concrete requirements
- repetitive scaffolding
- test skeletons
- mechanical refactors
- verification loops (`pnpm test`, `pnpm typecheck`, archive flow)

### What I decided myself

I kept these decisions explicit and reviewable:
- SQLite vs Postgres for the assessment
- raw SQL vs ORM
- sliding 7-day semantics
- service/router layering
- what stayed intentionally out of scope for the assessment

This repo was developed with a spec-driven workflow using:

- Pi: `https://pi.dev`

Typical flow:

2. Start a change with `/opsx-propose`
3. Implement with `/opsx-apply`
4. Verify with tests/typecheck
5. Archive with `openspec archive <name> -y`
6. Export the AI session into `ai-sessions/`
